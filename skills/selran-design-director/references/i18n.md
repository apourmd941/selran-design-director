# Internationalization — scripts, RTL, copy-length tolerance

The skill ships Western-first defaults (Latin script, LTR, English-length copy). The i18n layer makes that configurable so the same design system can ship in Arabic, Japanese, Russian, or Hindi without breaking. This is where most AI-generated design silently fails — English-first assumptions bake into font choices, button widths, and icon direction.

---

## The `i18n` schema block

```yaml
i18n:
  default_locale: "en"
  scripts: ["latin", "cjk", "arabic"]    # which scripts must work
  rtl: true                               # Arabic → layout flips
  max_expansion: 1.4                      # copy can grow 40% over English
```

Omit the block entirely for English-only / LTR-only projects.

## Three dimensions to configure

1. **Scripts** — which writing systems must render correctly (font coverage)
2. **Directionality** — whether layout must mirror (logical properties)
3. **Copy length** — how much the UI must stretch for non-English text

Each dimension is independent. You can support CJK without RTL (Japanese). You can need RTL without CJK (Arabic). You can need long copy without either (German).

## Dimension 1: Script coverage & font fallbacks

Every direction's display/body font covers Latin. None natively cover CJK, Arabic, Devanagari, or Hebrew well. When `i18n.scripts` includes anything beyond Latin, the build must chain fallback fonts.

The full mapping lives in `assets/i18n/font-fallbacks.yaml` — keyed by script and direction, so editorial's CJK fallback is a refined serif (Source Han Serif) while technical-minimal's is a clean sans (Noto Sans CJK).

### How to apply

**Web:**
```css
:root {
  --font-display: "General Sans", "Noto Sans", "Noto Sans CJK JP",
                  "Noto Sans Arabic", "Noto Sans Devanagari", sans-serif;
  --font-body: "General Sans", "Noto Sans", "Noto Sans CJK JP",
               "Noto Sans Arabic", system-ui, sans-serif;
}
```

Always put the direction's chosen face first, then the script-specific fallback for each supported script, ending with a generic family.

**iOS (SwiftUI):** Use `Font.custom` for the primary face, fall back to `.system` — iOS auto-resolves script-appropriate faces from the system stack. Load Noto fonts via `UIFont.fontDescriptor.withCascadeList` for granular control.

**Android (Compose):** Use `FontFamily` chained with a list. For CJK/Arabic/Devanagari, include `Noto Sans <Script>` from Google Fonts or bundle locally.

### Script-specific gotchas

- **CJK:** punctuation is full-width; line-height must increase ~1.7 for Japanese to breathe. Don't apply letter-spacing > 0 to CJK — it visibly separates ideographs.
- **Arabic:** must join; never apply `text-transform: uppercase` (no case concept). Numerals may be Western Arabic (0–9) or Eastern Arabic (٠–٩) depending on locale — respect `font-feature-settings: 'lnum'` or locale-aware formatting.
- **Devanagari:** line-height must increase ~1.8 to accommodate matras (vowel marks above/below the base line).
- **Hebrew:** RTL but not connected like Arabic. Similar layout flip, different font needs.
- **Thai:** no word boundaries in the script; `word-break: break-word` is incorrect. Use `word-break: keep-all` with locale-aware line-breaking.

## Dimension 2: RTL — layout mirroring

When `i18n.rtl: true`, layouts must mirror for Arabic/Hebrew. Mirror rules per component live in `assets/i18n/rtl-mirror.yaml`.

### Use logical properties, not physical

**Web:**
```css
/* WRONG — physical, doesn't flip */
.card { padding-left: 16px; border-right: 1px solid var(--border); }

/* RIGHT — logical, flips automatically in RTL */
.card { padding-inline-start: 16px; border-inline-end: 1px solid var(--border); }
```

Declare `dir="rtl"` on the `<html>` element when the locale is RTL. Logical properties do the rest.

**SwiftUI:** `.padding(.leading, 16)` → flips automatically with `layoutDirection`. Respect `@Environment(\.layoutDirection)` when positioning custom shapes.

**Compose:** `Modifier.padding(start = 16.dp)` → auto-flips. `LocalLayoutDirection.current` for manual reads.

### What to flip

| Element | Flip? | Why |
|---|---|---|
| Text alignment | Yes | Start-aligned text follows reading direction |
| Layout order (nav, tabs, breadcrumbs) | Yes | Reading order mirrors |
| Padding/margin `start/end` | Yes | Logical |
| Arrow icons (back, forward, chevron-right) | Yes | Direction-meaningful |
| Carousel next/prev | Yes | Follows reading direction |
| Progress bars | Yes | Fill direction reverses |
| Media controls (play, pause) | **No** | Universal conventions |
| Clock, calendar, phone icons | **No** | Not direction-meaningful |
| Logos | **No** | Brand is brand |
| Code blocks, URLs, mono text | **No** | Content is LTR even in RTL layout |
| Numerical axes in charts | Depends | Left-to-right is common; consult locale |

See `assets/i18n/rtl-mirror.yaml` for the component-by-component list.

### Bidirectional content

Arabic UI with an English URL, or Hebrew body with an English brand name — these are common. Use `<bdi>` or `unicode-bidi: isolate` to keep the embedded LTR run from corrupting the surrounding RTL flow:

```html
<p lang="ar" dir="rtl">
  تحقق من موقعنا <bdi>example.com</bdi> للمزيد من المعلومات.
</p>
```

## Dimension 3: Copy-length tolerance

Non-English copy runs longer. This is the single most common cause of broken UI on international launches.

| Language | Expansion vs. English | Worst case |
|---|---|---|
| German | 1.3× | Compound nouns ("Geschwindigkeitsbegrenzung") |
| Russian | 1.2× | Cyrillic runs wide |
| Spanish | 1.2× | Verb constructions |
| French | 1.2× | Prepositions |
| Arabic | 1.25× | Words join; numerals don't expand |
| Japanese | 0.7× | Ideographs are dense (usually shorter) |
| Korean | 0.9× | Between CJK and Latin |
| Finnish | 1.6× | Agglutinative — "alkoholiohjelmisto" |
| Portuguese | 1.3× | Verbal constructions |

Set `i18n.max_expansion` to the worst case among your target locales. `1.4` covers most Western + Asian needs; `1.6` covers Finnish/Hungarian.

### Where expansion breaks things

- **Buttons:** "Save" (4ch) → "Speichern" (9ch). Fixed-width buttons truncate. Always `padding: <pad> <pad>; width: auto`.
- **Nav items:** horizontal nav bars run out of space. Either allow wrap to 2 lines, or switch to a hamburger earlier.
- **Table headers:** "Date" → "Datum"; "Amount" → "Betrag". Tables need flexible column widths with `min-width` for numeric columns only.
- **Form labels:** "Email" (5ch) → "Correo electrónico" (17ch). Stacked labels (above field) handle this; inline labels break.
- **Tabs:** long labels force overflow-scroll or reflow to a dropdown.
- **Tooltips:** tooltips that just-fit in English overflow off-screen in German. Cap `max-width` and allow wrap.

### Overflow strategy per direction

| Direction | Strategy |
|---|---|
| **technical-minimal** | Truncate with ellipsis; tooltip on hover to reveal full text. Dense UI. |
| **editorial** | Wrap to 2 lines; adjust leading to keep rhythm. |
| **bold-distinctive** | Reduce display-size by one step if a headline would overflow. Don't shrink type below `xxl`. |
| **dark-premium** | Truncate with ellipsis; hover reveals. |
| **warm-approachable** | Wrap freely; conversational UI tolerates 2–3 line labels. |
| **vibrant-playful** | Wrap; motion softens the reflow. |
| **brutalist** | Let it break the grid. Brutalism tolerates overflow as a feature. |

Never truncate critical information (error messages, required field labels, CTAs). Truncate metadata only (secondary text, captions, tag labels).

## Numbers, dates, currency

Use the platform's locale-aware formatters, not string templates:

- **Web:** `Intl.NumberFormat`, `Intl.DateTimeFormat`, `Intl.PluralRules`
- **iOS:** `NumberFormatter`, `DateFormatter` with `.locale`
- **Android:** `NumberFormat.getInstance(locale)`, `SimpleDateFormat` with `Locale`

Never hardcode thousand separators, decimal points, or date formats. `1,234.56` is US; `1.234,56` is German; `1 234,56` is French with non-breaking space.

Tabular numerals (from `personality` gestures in several directions) still apply to all scripts that use Arabic numerals (0–9). Eastern Arabic numerals (٠–٩) also benefit from tabular alignment in CSS (`font-feature-settings: 'tnum'` works on either numeral set if the font supports it).

## Icons and imagery

- **Directional icons** (arrows, chevrons, undo/redo, sent/received messages) flip under RTL.
- **Non-directional icons** (gear, home, bell, avatar placeholder) don't flip.
- **Photography:** avoid culturally specific imagery as hero content unless the target locale matches. A wheat-field hero in the Middle East, a hand gesture with "thumbs up" in regions where it's offensive, etc.
- **Emoji:** rendering differs per platform and locale. Don't use emoji as UI icons (already covered in `imagery.md`) — especially not in i18n builds.

## The a11y overlap

RTL and i18n pair with accessibility:

- `<html lang="ar" dir="rtl">` — set both
- `aria-label` must be localized, not English
- Screen readers respect the lang attribute to switch voices (Arabic voice for Arabic text)
- Keyboard navigation: `Tab` order follows DOM order; in RTL this means right-to-left visual flow
- Focus indicators don't change — same outline, same contrast requirements

## Testing checklist

Before shipping a multi-locale build:

- [ ] Render a screen in each target locale using realistic copy (not "Lorem ipsum translated")
- [ ] Test the worst-case expansion language (Finnish/German)
- [ ] Test an RTL locale — buttons, icons, text alignment, arrow directions
- [ ] Test a CJK locale — line-height, punctuation, character rendering
- [ ] Verify locale-aware number/date/currency formatting
- [ ] Verify `lang` attribute is set correctly
- [ ] Verify no hardcoded string concatenation (`"You have " + n + " items"` — use ICU MessageFormat)

## Anti-patterns

- **"We'll localize later."** Adds 30% of the UI work back in rework. Declare `i18n` at Phase 2 if you know locales are coming.
- **Hardcoded English strings.** Every user-facing string must be a translatable key, not inline text.
- **Reusing the same key for different contexts.** "Save" as a verb (button) and "Save" as a noun (savings, in finance UI) translate differently.
- **Assuming English word order.** "Delete 3 items?" → Russian requires plural form depending on the number (3 vs 5 vs 21). Use ICU plural forms.
- **Flipping logos, brand names, or code.** Some things stay LTR even in RTL layouts.
- **Using flags to represent languages.** Languages ≠ countries. Spanish isn't only Spain. Use language names in the native script ("العربية", not a Saudi flag).
